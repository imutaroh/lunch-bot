# 用語集 (発表で出てくる全用語)

各用語は 4 角度で書く。空欄禁止。
- ① 一言で (30字以内)
- ② lunch-bot で言うと
- ③ 日常の比喩で (料理・住所・郵便・部屋など。他言語との比較は禁止)
- ④ もう一つの言い方 (誰がいつ使うかなど)

## 全体構造の用語

### handler

- **① 一言で**: リクエストを受け取って service に橋渡しする入口の係
- **② lunch-bot で言うと**: `internal/handler/lunch_handler.go`。`Recruit()` `Announce()` の 2 メソッドだけの薄い層
- **③ 日常の比喩で**: 受付カウンター。お客さんから注文を受けて、奥の調理場 (service) に伝票を回すだけ。料理は作らない
- **④ もう一つの言い方**: 入口の形式 (CLI コマンド名 / HTTP リクエスト) を吸収して、業務ロジックには影響を与えない壁

### service

- **① 一言で**: 業務ロジック本体。アプリの「やりたいこと」を書く層
- **② lunch-bot で言うと**: `internal/service/lunch_service.go`。シャッフル、リアクション集計、メッセージ組立はここ
- **③ 日常の比喩で**: 調理場。受付 (handler) からきた伝票を見て、材料 (repository) を取り寄せて料理を作る場所
- **④ もう一つの言い方**: handler と repository の間で「何をするか」を決めている真ん中の層

### repository

- **① 一言で**: 外部とのやり取りを担当する係 (API 呼び出し / DB アクセス)
- **② lunch-bot で言うと**: `internal/repository/slack_client.go`。Slack API を叩く全コードがここに集約
- **③ 日常の比喩で**: 食材の仕入れ係。市場 (Slack API) に行って材料を取ってきて、調理場 (service) に渡す
- **④ もう一つの言い方**: 「外の世界」と話せる唯一の層。ここを差し替えれば fake にも本物にもできる

### 3層アーキテクチャ

- **① 一言で**: handler / service / repository の 3 層に分けた構造
- **② lunch-bot で言うと**: `internal/handler/` `internal/service/` `internal/repository/` の 3 ディレクトリそのまま
- **③ 日常の比喩で**: レストランの「受付 / 厨房 / 仕入れ」の 3 部屋構成。それぞれ専門が違う
- **④ もう一つの言い方**: 「入口」「中身」「外」を分ける定番の設計パターン

### 責務分離

- **① 一言で**: 各層が「自分の仕事だけ」を持ち、他の層の仕事に踏み込まない設計方針
- **② lunch-bot で言うと**: handler は Slack を叩かない、repository はシャッフルしない。各層が役割を守る
- **③ 日常の比喩で**: 受付の人がレジも調理もしないこと。役割が混ざらないので、片方だけ変えても残りに影響しない
- **④ もう一つの言い方**: Single Responsibility Principle の実践版

## Go の言葉

### struct

- **① 一言で**: フィールド (データ) をまとめた箱
- **② lunch-bot で言うと**: `LunchService` `SlackClient` `Config` などはすべて struct
- **③ 日常の比喩で**: お弁当箱。仕切りごとにご飯・おかず・果物を入れる。中身は決まっている
- **④ もう一つの言い方**: 関連するデータを 1 つに束ねる定義。インスタンスを作るときの「型」

### receiver

- **① 一言で**: メソッドが「どの struct のものか」を示す紐づけ
- **② lunch-bot で言うと**: `func (s *LunchService) RunRecruit() error` の `(s *LunchService)` の部分
- **③ 日常の比喩で**: 名刺の所属欄。「この技は LunchService の技ですよ」と書いてあるイメージ
- **④ もう一つの言い方**: メソッドを struct に紐付ける書き方。これがないとただの関数

### interface

- **① 一言で**: 「これらのメソッドを持っていれば誰でも OK」という契約書
- **② lunch-bot で言うと**: `SlackRepository` interface。`PostMessage` などのメソッドさえあれば本物でも fake でも受け入れる
- **③ 日常の比喩で**: 求人票。「料理が作れる人なら誰でもいい」と書いてあって、応募条件さえ満たせば採用される
- **④ もう一つの言い方**: 「実装の中身」ではなく「できること」だけを定義した型。差し替え可能性の根本

### DI (Dependency Injection)

- **① 一言で**: 依存先を「外から渡してもらう」設計手法
- **② lunch-bot で言うと**: `cmd/bot/main.go` で `NewLunchService(slack, ch)` のように、必要な部品を引数で渡している箇所
- **③ 日常の比喩で**: 料理人が市場まで自分で行かず、「食材は誰かが届けてくれる」前提で料理する。届ける人を変えるだけで料理人のレシピは変わらない
- **④ もう一つの言い方**: 「何を使うか」を呼び出す側に決めさせること。テスト時に fake に差し替えられる仕組みの土台

### コンストラクタ

- **① 一言で**: struct を初期化して返す関数 (Go では `NewXxx`)
- **② lunch-bot で言うと**: `NewLunchService(slack, ch)` `NewSlackClient(token)` がそれ
- **③ 日常の比喩で**: 工場の組み立てライン。部品 (引数) を渡すと完成品 (struct) が出てくる
- **④ もう一つの言い方**: 「正しく初期化された struct」を保証する入口関数。直接 `&LunchService{}` するより安全

### fake (モック)

- **① 一言で**: 本物の代わりに使う偽物の実装
- **② lunch-bot で言うと**: `cmd/simulate` の `fakeSlack` 構造体。Slack を叩かずメモリ上で振る舞うだけ
- **③ 日常の比喩で**: 演劇のリハーサルで使う「本物そっくりの小道具」。本番ではないから壊しても大丈夫
- **④ もう一つの言い方**: interface を満たす最低限のダミー実装。テストや動作確認で本番依存を切り離すために使う

### fmt.Errorf %w / wrap

- **① 一言で**: 元のエラーに文脈を足して再送出する書き方
- **② lunch-bot で言うと**: `return fmt.Errorf("post recruitment: %w", err)` のような書き方が全層に出てくる
- **③ 日常の比喩で**: 不良品を上司に報告するときに、元の不良票はそのまま貼り付けて、自分の状況メモを上に乗せる
- **④ もう一つの言い方**: エラーの「原因の連鎖」を保ったまま積み重ねる手法。`errors.Is/As` で元のエラーを取り出せる

## ルーティング・HTTP の言葉

### chi / chiルータ

- **① 一言で**: Go で HTTP のパスを関数に紐付けるためのライブラリ
- **② lunch-bot で言うと**: lunch-bot は HTTP サーバを持たないので使っていない (CLI のため)
- **③ 日常の比喩で**: 郵便局の仕分け係。宛先 (URL) を見て、正しい部屋 (handler) に手紙を届ける
- **④ もう一つの言い方**: `r.Post("/customers", h.PostCustomers)` のように URL とメソッドを定義する道具。Polaris の `cmd/api/routes.go` で大活躍

### HTTP

- **① 一言で**: Web 上でクライアントとサーバが会話するためのルール
- **② lunch-bot で言うと**: `repository/slack_client.go` が `http.Client` で Slack API を叩いている部分
- **③ 日常の比喩で**: 郵便。住所 (URL) に手紙 (リクエスト) を送ると返事 (レスポンス) が来る
- **④ もう一つの言い方**: リクエスト / レスポンス形式の通信プロトコル。Web の根幹

### HTTPメソッド (POST / GET)

- **① 一言で**: HTTP リクエストの「目的」を表す動詞 (取る / 送る など)
- **② lunch-bot で言うと**: `chat.postMessage` を `POST` で叩いている箇所
- **③ 日常の比喩で**: 郵便の種類。GET = 「資料請求してください」/ POST = 「申込書を提出します」
- **④ もう一つの言い方**: GET は「取得」、POST は「送信・登録」が代表的な使い方

### エンドポイント

- **① 一言で**: HTTP リクエストを受け付ける URL の入口
- **② lunch-bot で言うと**: lunch-bot は HTTP を受けないので持っていない (CLI のみ)
- **③ 日常の比喩で**: お店の窓口。「弁当注文窓口」「クレーム窓口」のように、用途別に窓口がある
- **④ もう一つの言い方**: `/api/v2/ecforce/customers` のような URL パス + メソッドのペア

### URL

- **① 一言で**: Web 上のリソースを特定するための住所
- **② lunch-bot で言うと**: Slack API の `https://slack.com/api/chat.postMessage` がそれ
- **③ 日常の比喩で**: 郵便の宛先。「東京都千代田区...」のように、世界で 1 つに特定できる住所
- **④ もう一つの言い方**: スキーム (https) + ホスト (slack.com) + パス (/api/...) の組み合わせ

### 認証

- **① 一言で**: 「あなたが本人ですよ」をサーバに証明する仕組み
- **② lunch-bot で言うと**: `Bearer xoxb-...` トークンを HTTP ヘッダに付けて Slack に送っている部分
- **③ 日常の比喩で**: 会員カードを見せて「会員ですよ」と証明する。カードがないと入れない店
- **④ もう一つの言い方**: トークン / OIDC / API キー など、形式は色々。Polaris では OIDC token を多用

## Cloud (GCP) の言葉

### Cloud Run

- **① 一言で**: コンテナ化した Go バイナリを HTTP サーバとして動かす GCP のサービス
- **② lunch-bot で言うと**: lunch-bot は使っていない (GitHub Actions で直接 go run している)。Polaris の unified-api はここで常駐している
- **③ 日常の比喩で**: 借りたキッチン。料理人 (バイナリ) が来店するたびに調理して、注文がなければ閉まる
- **④ もう一つの言い方**: HTTP リクエストが来たときだけ起動して動く、サーバ管理不要の実行環境

### Cloud Run Jobs

- **① 一言で**: Cloud Run のバッチ版。HTTP を持たず、CLI を 1 回走らせて終わるモード
- **② lunch-bot で言うと**: lunch-bot を `go run ./cmd/bot recruit` で動かすのと同じ感覚を、クラウド上でやる版
- **③ 日常の比喩で**: 出張シェフ。「30 分後にこの料理を作って終わったら帰る」と決めて呼ぶサービス
- **④ もう一つの言い方**: Polaris の `cmd/jobs/main.go` を起動するための実行環境。長時間バッチ用の脱出口

### Cloud Functions

- **① 一言で**: 関数 1 つだけをデプロイして動かす GCP のサービス
- **② lunch-bot で言うと**: 使っていない。Polaris の旧世代 ingestion (Python) はこれで動いていた
- **③ 日常の比喩で**: 1 品料理だけ作る屋台。「唐揚げだけ売ってる」みたいな単機能の店
- **④ もう一つの言い方**: 軽い処理向け。Polaris では新世代の unified-api (Cloud Run) に置き換えられつつある

### Cloud Scheduler

- **① 一言で**: 時間が来たら HTTP を叩いてくれるクラウドのタイマー
- **② lunch-bot で言うと**: GitHub Actions の cron がやっている役割と同じ
- **③ 日常の比喩で**: 目覚まし時計。「毎朝 7 時に郵便配達を呼ぶ」と設定しておけば、自分で起きなくていい
- **④ もう一つの言い方**: cron 式 (`0 0 * * *` 等) で定期実行を設定する GCP の標準ツール

### cron

- **① 一言で**: 「毎日 9 時」のような定期実行スケジュールの書き方
- **② lunch-bot で言うと**: `.github/workflows/recruit.yml` の `cron: '0 0 * * MON,TUE'` 部分
- **③ 日常の比喩で**: 月曜の朝にゴミ収集車が来る、のような決まった曜日・時刻の予定
- **④ もう一つの言い方**: `分 時 日 月 曜日` の 5 個の数字でスケジュールを表す Unix 由来の記法

### Cloud Tasks

- **① 一言で**: 「あとでこの HTTP を叩いてね」と頼める待ち行列サービス
- **② lunch-bot で言うと**: 持っていない。lunch-bot は同期 1 段なので、間に置く必要がない
- **③ 日常の比喩で**: 出前注文を受け付ける配達アプリ。注文を積んでおけば、配達員が個別に運んでくれる。1 件失敗しても他は進む
- **④ もう一つの言い方**: 非同期 HTTP 呼び出しのキュー。リトライ・並列実行・遅延配送を任せられる

### キュー

- **① 一言で**: 「順番待ちの行列」を保持する仕組み
- **② lunch-bot で言うと**: 持っていない (同期 1 段)
- **③ 日常の比喩で**: 銀行の番号札。先に来た人から順に呼ばれる。途中で 1 人いなくても次に進める
- **④ もう一つの言い方**: 入れた順 (FIFO) に取り出される箱。Cloud Tasks の本体はキュー

### ファンアウト

- **① 一言で**: 1 件の入力を N 件のジョブに展開すること
- **② lunch-bot で言うと**: 持っていない。lunch-bot は 1 cron = 1 ジョブで終わる
- **③ 日常の比喩で**: 1 通の本部指令を、支店ごとに分けて配るイメージ。「全店で在庫確認を始めろ」を 50 店舗分のメモに変える
- **④ もう一つの言い方**: 1 入力 → N 出力に分散すること。task-dispatcher が「1 cron → N テナント分のジョブ」に展開するのが代表例

### リトライ

- **① 一言で**: 失敗したら自動で再試行すること
- **② lunch-bot で言うと**: 持っていない (失敗したら GitHub Actions のワークフロー全体を再実行する手段しかない)
- **③ 日常の比喩で**: 通話が切れたら自動でかけ直してくれる電話。3 回までかけて諦める設定もできる
- **④ もう一つの言い方**: Cloud Tasks がタスク単位で勝手にやってくれる。回数や間隔を指定可能

### 非同期

- **① 一言で**: 結果を待たずに次に進むやり方
- **② lunch-bot で言うと**: 持っていない (lunch-bot は完全同期)
- **③ 日常の比喩で**: 出前注文。「注文した瞬間にあなたの仕事は終わり、配達は別の人がやる」
- **④ もう一つの言い方**: 呼び出し元と呼び出し先のタイミングが切れている状態。Cloud Tasks 経由はすべて非同期

### 同期

- **① 一言で**: 結果が返ってくるまで待つやり方
- **② lunch-bot で言うと**: lunch-bot 全体が同期。`PostMessage` を呼んだら投稿完了まで待つ
- **③ 日常の比喩で**: 店頭で料理を頼んで、できあがるまで席で待つ
- **④ もう一つの言い方**: 呼び出し元がブロック (待機) する形。シンプルだが時間がかかる処理に弱い

### OIDC token

- **① 一言で**: Google が発行する「お前は本物のサービスだ」と証明する身分証
- **② lunch-bot で言うと**: 持っていない (GitHub Actions の Secret で代用)
- **③ 日常の比喩で**: 区役所が発行する身分証明書。「この人は本物の住民です」と国が保証する形
- **④ もう一つの言い方**: Cloud Scheduler や Cloud Tasks が「俺は正規の呼び出し元だ」と Cloud Run に証明するときに付けるトークン

## Polaris のコンポーネント

### Polaris

- **① 一言で**: 複数 EC・広告 API からデータを集めて BigQuery に貯める社内 ETL 工場
- **② lunch-bot で言うと**: lunch-bot を 100 倍規模・複数連携先・非同期化したもの
- **③ 日常の比喩で**: 大型物流センター。色んな仕入れ先から荷物が届いて、仕分けして、倉庫に並べて、出荷する施設
- **④ もう一つの言い方**: ingestion + orchestration + transformation + application で構成された、社内データ基盤

### task-dispatcher

- **① 一言で**: Scheduler から呼ばれて Cloud Tasks に「テナント別ジョブ」を積む係
- **② lunch-bot で言うと**: GitHub Actions の cron + `cmd/bot/main.go` のサブコマンド分岐部分を、テナント数だけ展開する版
- **③ 日常の比喩で**: 配達指示センター。本部 (Scheduler) からの指令を受けて、店舗ごとに伝票を切って配達待ち箱 (Cloud Tasks) に放り込む
- **④ もう一つの言い方**: `orchestration/task-dispatcher/cmd/server/main.go` で動く Go の HTTP サーバ。22 個の `/dispatch/<platform>` を持つ

### unified-api

- **① 一言で**: HTTP/CLI で起動されてデータを集め BigQuery に書く Polaris の本体
- **② lunch-bot で言うと**: lunch-bot の handler / service / repository をそのまま大規模化したもの
- **③ 日常の比喩で**: 物流センター内の倉庫部隊。指示書 (HTTP リクエスト) を受けて、外部から品物を取り寄せて、棚 (BigQuery) に置く
- **④ もう一つの言い方**: `ingestion/unified-api/` 配下で動く Go の HTTP サーバ + CLI バッチ。17 種類の連携先に対応

### ingestion / orchestration / transformation

- **① 一言で**: Polaris の 3 大フォルダ。集める / 制御する / 加工する
- **② lunch-bot で言うと**: lunch-bot の `internal/repository` (集める) + `internal/service` (加工) + GitHub Actions (制御) を 3 つのフォルダに分けた版
- **③ 日常の比喩で**: 物流センターの「入荷部 / 配車部 / 加工部」。それぞれ別チームが回している
- **④ もう一つの言い方**: ingestion = データ取得、orchestration = 実行制御、transformation = データ加工

## データ・ストレージの言葉

### BigQuery

- **① 一言で**: GCP の超大容量データウェアハウス (DWH)
- **② lunch-bot で言うと**: lunch-bot は Slack 自体がデータソースなので持っていない
- **③ 日常の比喩で**: 巨大な業務用倉庫。色んな業者の荷物を全部ここに集めて、後で必要に応じて取り出せる
- **④ もう一つの言い方**: SQL で読み書きできるクラウド DB。dbt の主戦場でもある

### DWH (データウェアハウス)

- **① 一言で**: 分析用にデータを集約・整理する大型データ倉庫
- **② lunch-bot で言うと**: 持っていない
- **③ 日常の比喩で**: 図書館。色んな本 (データ) が分類されて並んでいて、目的別に検索できる
- **④ もう一つの言い方**: BigQuery / Snowflake / Redshift などが代表的な実装

### dataset

- **① 一言で**: BigQuery 内のテーブルをまとめるフォルダのような単位
- **② lunch-bot で言うと**: 持っていない
- **③ 日常の比喩で**: 倉庫の中の「区画」。「Amazon 区画」「楽天 区画」のように荷物を分けて置く
- **④ もう一つの言い方**: テーブルの上位概念。Polaris では `Raw_*` `Hrd_*` などの命名規則で 4 段階を表す

### staging

- **① 一言で**: dbt の最初の層。生データの型と命名を整える
- **② lunch-bot で言うと**: 持っていない (lunch-bot に dbt はない)
- **③ 日常の比喩で**: 仕入れた食材の「下ごしらえ」。皮をむいて、サイズを揃えるだけ。まだ料理ではない
- **④ もう一つの言い方**: `Hrd_*` プレフィックス。マクロ 1 行で量産されることが多い

### intermediate

- **① 一言で**: dbt の 2 層目。抽出 → 結合 → 統合 → ビジネスルールの段階加工
- **② lunch-bot で言うと**: 持っていない
- **③ 日常の比喩で**: 仕込み。下ごしらえした食材を組み合わせて、煮物の素を作る段階
- **④ もう一つの言い方**: `1_extract / 2_join / 3_union / 4_soft-business / 99_transform` の番号付きフォルダで処理順を可視化

### dwh (層名としての)

- **① 一言で**: dbt の 3 層目。全社共通の整形済み倉庫
- **② lunch-bot で言うと**: 持っていない
- **③ 日常の比喩で**: 仕込みが終わった料理を保管する冷蔵庫。いつでも出せる状態
- **④ もう一つの言い方**: bdw / rdw / ref などの内部分類があるが発表では深入りしない

### mart

- **① 一言で**: dbt の最終層。BI / レポート / 書き戻し用の最終形
- **② lunch-bot で言うと**: 持っていない
- **③ 日常の比喩で**: 提供。お客さんに出す皿に盛りつけた料理
- **④ もう一つの言い方**: `dim_*` (ディメンション) と `fct_*` (ファクト) の組み合わせ = スタースキーマ

### dbt

- **① 一言で**: BigQuery 内で SQL を順番に走らせて加工するツール
- **② lunch-bot で言うと**: 持っていない (in-memory でシャッフルして終わる)
- **③ 日常の比喩で**: 工場の自動加工ライン。原材料を順番にラインに流すと、最終製品になって出てくる
- **④ もう一つの言い方**: SQL + YAML だけで動く別世界。Polaris の `transformation/androots/` がそれ

### SQL

- **① 一言で**: データベースに「こういうデータが欲しい」と頼む言語
- **② lunch-bot で言うと**: 使っていない
- **③ 日常の比喩で**: 図書館の検索カード。「夏目漱石の本で、1900 年以降のもの」のように条件を書いて投げる
- **④ もう一つの言い方**: `SELECT ... FROM ... WHERE ...` の形式。dbt も BigQuery も SQL で動く

### マクロ

- **① 一言で**: dbt 内で再利用される SQL のテンプレート関数
- **② lunch-bot で言うと**: 持っていない
- **③ 日常の比喩で**: 料理レシピのテンプレート。「炒め物の基本手順」を 1 枚作っておけば、具材を変えて何回でも使える
- **④ もう一つの言い方**: staging の SQL を「マクロ呼び出し 1 行」に圧縮するために使われる

### スタースキーマ

- **① 一言で**: ファクト (事実) を中心に、ディメンション (属性) を周りに置く DWH の定番構造
- **② lunch-bot で言うと**: 持っていない
- **③ 日常の比喩で**: 売上日記 (中心) に、日付・商品・店舗 (周辺) の情報がぶら下がっている形
- **④ もう一つの言い方**: BI ツールが集計しやすい形。中心が `fct_*`、周辺が `dim_*`

### dim (ディメンション)

- **① 一言で**: スタースキーマの「属性側」のテーブル
- **② lunch-bot で言うと**: 持っていない
- **③ 日常の比喩で**: 商品カタログ・カレンダーなど、「説明書き」にあたる情報
- **④ もう一つの言い方**: `dim_date` `dim_product` のような命名。事実テーブルと結合して使う

### fct (ファクト)

- **① 一言で**: スタースキーマの「事実側」のテーブル
- **② lunch-bot で言うと**: 持っていない
- **③ 日常の比喩で**: 取引履歴・売上記録など、「いつ・誰が・何を・いくらで」の事実
- **④ もう一つの言い方**: `fct_orders` `fct_sales` のような命名。dim と結合して分析する

### Supabase

- **① 一言で**: テナント設定を保管するマスタデータ DB (Polaris で使用)
- **② lunch-bot で言うと**: `internal/config/config.go` の役割を Web UI 付き DB にした感じ
- **③ 日常の比喩で**: 顧客名簿。誰がどの店舗の担当か、どんな設定で動くかを管理している台帳
- **④ もう一つの言い方**: Postgres ベースのオープンソース BaaS。Polaris では `mdm` と `mdm_new` の 2 系統がある

### マスタデータ / mdm

- **① 一言で**: 「テナント一覧」「商品一覧」など、業務の土台となる基準データ
- **② lunch-bot で言うと**: `internal/config/config.go` の環境変数読込部分が一番近い (チャネル ID など)
- **③ 日常の比喩で**: お店の会員リスト。誰がいて何が好きで、何回来店したかを束ねた台帳
- **④ もう一つの言い方**: Master Data Management の略。Polaris では Supabase に置かれている

### テナント

- **① 一言で**: Polaris で扱う「1 つの顧客 (会社)」の単位
- **② lunch-bot で言うと**: 持っていない (lunch-bot はチャンネル 1 個だけ)
- **③ 日常の比喩で**: マンションの 1 部屋。1 つの建物 (Polaris) に複数の住人 (顧客) が入っていて、部屋ごとに別契約
- **④ もう一つの言い方**: 1 つの cron が走るたびに、テナント数だけのジョブにファンアウトされる単位

## ETL の言葉

### ETL (Extract / Transform / Load)

- **① 一言で**: データを取って・変えて・戻す処理パターンの総称
- **② lunch-bot で言うと**: lunch-bot 全体が小さな ETL (Slack 取得 → シャッフル → Slack 投稿)
- **③ 日常の比喩で**: 仕入れ → 調理 → 提供 の 3 段階。料理屋の基本フロー
- **④ もう一つの言い方**: データエンジニアリングの基本パターン。Polaris は大規模 ETL

### Extract (取る)

- **① 一言で**: 外部からデータを引っ張ってくる工程
- **② lunch-bot で言うと**: `repository.GetReactions` で Slack のリアクション一覧を取得する箇所
- **③ 日常の比喩で**: 市場での仕入れ。野菜屋から白菜を買ってくる工程
- **④ もう一つの言い方**: 外部 API / DB / ファイルから読み出す段階

### Transform (変換)

- **① 一言で**: 取ってきたデータを使いやすい形に変える工程
- **② lunch-bot で言うと**: service の `shuffle()` でリアクション者をグループに分ける箇所
- **③ 日常の比喩で**: 下ごしらえ + 調理。仕入れた食材を皿に盛れる形に変える
- **④ もう一つの言い方**: 計算・整形・結合・集計など、データを変形する段階

### Load (戻す)

- **① 一言で**: 変換したデータを行き先に書き込む工程
- **② lunch-bot で言うと**: `PostMessage` で Slack に発表メッセージを投稿する箇所
- **③ 日常の比喩で**: 提供。完成した料理を客の前に出す工程
- **④ もう一つの言い方**: BQ / Slack / 外部システムへの書き込み段階

## ジョブ / API 呼び出しの言葉

### API エンドポイント呼び出し

- **① 一言で**: unified-api の HTTP エンドポイント (`/api/v2/...`) を叩くこと
- **② lunch-bot で言うと**: lunch-bot は HTTP サーバを持たないので持っていない
- **③ 日常の比喩で**: 店の窓口に直接行って注文する形。窓口 (handler) が伝票を受け取る
- **④ もう一つの言い方**: Cloud Run の常駐サーバが受ける入口。短時間で終わる処理向け

### ジョブ呼び出し

- **① 一言で**: Cloud Tasks 経由で API エンドポイントを **非同期に** 叩くこと
- **② lunch-bot で言うと**: 持っていない (lunch-bot は同期 1 段)
- **③ 日常の比喩で**: 出前注文。「あとでこの料理を作ってこの住所に届けて」を出前アプリに登録する
- **④ もう一つの言い方**: task-dispatcher が Cloud Tasks に積むタスク = ジョブ。リトライ・並列実行が付いてくる

### runner

- **① 一言で**: ジョブの実行本体 (`internal/jobs/<platform>/runner.go` 等)
- **② lunch-bot で言うと**: `cmd/bot/main.go` のサブコマンド分岐内で `h.Recruit()` を呼ぶ部分が近い
- **③ 日常の比喩で**: 出張先で実際に料理を作る人。レシピ (service) は同じだが、店ではなく現場で動く
- **④ もう一つの言い方**: handler を経由せず、CLI 経由で service を直接叩く層。Cloud Run Jobs 用

### 失敗の局所化

- **① 一言で**: 1 つのタスクが失敗しても、他のタスクは影響を受けない設計
- **② lunch-bot で言うと**: 持っていない (1 つコケると全部失敗扱い)
- **③ 日常の比喩で**: 50 個の出前注文のうち 3 つが配達失敗しても、残り 47 は届く
- **④ もう一つの言い方**: Cloud Tasks 経由でジョブを積む唯一の利点の 1 つ。並列・個別リトライとセット

### タイムアウト

- **① 一言で**: 「これ以上待つのは諦める」時間制限
- **② lunch-bot で言うと**: `http.Client{Timeout: 10 * time.Second}` の 10 秒がそれ
- **③ 日常の比喩で**: 「30 分以内に来なかったら配達キャンセル」の制限時間
- **④ もう一つの言い方**: Cloud Run は 30 分が上限。だから長時間処理は Cloud Run Jobs に逃がす
