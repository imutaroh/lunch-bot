## Q1. なぜ lunch-bot を 3 層 (handler / service / repository) に分けたんですか?

- **レベル**: 簡単
- **聞かれそうな相手**: 笹本さん / 先輩
- **模範回答** (5秒以内に第一声、200字以内):
  > 「役割を分けると、片方を変えても残りに影響しないからです」。handler は入口の形式、service は業務ロジック、repository は外部 API、と仕事を分けています。例えば Slack を Discord に変えるなら repository だけ書き換えれば service は 1 文字も触らなくて済む。差し替えやすさと読みやすさのため、です。
- **L3 ポイント** (なぜそう答えるか):
  > 「責務分離」を抽象的に言うと伝わらないので、必ず lunch-bot の具体例 (Slack→Discord 差し替え) で着地させる。Polaris で repository が 17 個に増えても同じ理由が効く、と続けると深い。

## Q4. ジョブ呼び出しと API エンドポイント呼び出しの違いは?

- **レベル**: 中
- **聞かれそうな相手**: 笹本さん / 先輩
- **模範回答** (5秒以内に第一声、200字以内):
  > 「叩く経路と起動の仕方が違います」。API エンドポイント呼び出しは HTTP で `/api/v2/...` を叩く方、Cloud Run の常駐サーバが受ける、短時間処理向け。ジョブ呼び出しは Cloud Run Jobs で `cmd/jobs/main.go` を CLI 起動する方、handler を通らず直接 service を叩く、30 分超の長時間処理用の脱出口です。
- **L3 ポイント** (なぜそう答えるか):
  > 用語が混ざる質問なので、まず「経路と起動の仕方が違う」と一言で型を決めてから、それぞれ短く対比する。「service は同じものを使い回す」を最後に添えると interface の話と繋がる。

## Q5. lunch-bot と Polaris の一番大きな違いはどこですか?

- **レベル**: 中
- **聞かれそうな相手**: ノブローさん / 上司
- **模範回答** (5秒以内に第一声、200字以内):
  > 「規模じゃなくて『非同期かどうか』です」。lunch-bot は cron が直接 Go バイナリを叩いて完了まで待つ、同期 1 段。Polaris は cron が dispatcher を叩いた瞬間に dispatcher の仕事は終わり、本体は Cloud Tasks 経由で別プロセスが拾って動く、非同期 4 段。これにより 30 分超でも cron はタイムアウトしません。
- **L3 ポイント** (なぜそう答えるか):
  > 「規模が違います」と答えると浅く聞こえる。「規模じゃなく非同期」と置き換えると一段深く見える。スライド 12〜13 の核がそのまま使える。

## Q6. dbt は何屋ですか? 1 分で説明してください

- **レベル**: 中
- **聞かれそうな相手**: 笹本さん
- **模範回答** (5秒以内に第一声、200字以内):
  > 「BigQuery の中だけで SQL を順番に走らせて加工するツール」です。Go や Python ではなく、SQL と YAML だけで動く別世界。Polaris では unified-api が集めたデータを `Raw_*` に貯めて、そこから dbt が staging で型を整え、intermediate で組み立て、dwh で倉庫に置き、mart で BI 用に出す。料理工程と同じ 4 段階です。
- **L3 ポイント** (なぜそう答えるか):
  > 「Go の世界はここで終わる」と境界を引くと聞き手に優しい。深掘りされたら「マクロは SQL の関数で、staging は 1 行で済む」を 1 つだけ足す。

## 受け答えの大原則 (リハ前にもう一度確認)

1. **5 秒以内に第一声**: 「えーと」を捨てる。質問が来たら「〜です」「〜だからです」で即着地
2. **答えに迷ったら『一言の核 → 具体例 → 抽象化』** の順
3. **知らないことは「読んでません、次のステップです」と即答** (誤魔化しは一番危険)
4. **lunch-bot の具体ファイル名を必ず引く** (`shuffler_test.go` / `routes.go` / `cmd/simulate` など)
5. **Polaris との対応を最後に添える** (聞き手が一番興味あるのは業務との繋がり)
