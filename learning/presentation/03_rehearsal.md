## リハで詰まりやすい 3 スライド

### スライド 8 (対応表 ★)

- **詰まる理由**: 表が 7 行あり、180 秒で全部読もうとすると焦る
- **対策**: 台本通り「上から 3 行 (起動の引き金・入口層・業務) に集中」と最初に宣言する。残りは「次のスライド以降で深掘り」と流す。**全部を喋ろうとしない**
- **キラーフレーズ**: 「結論を一行で言うと、3 層プラス DI の骨格は完全に同じ。違いは 4 つだけ」

### スライド 11 (ジョブ vs API ★)

- **詰まる理由**: 「ジョブ呼び出し」「API エンドポイント呼び出し」「Cloud Run Jobs」「Cloud Tasks」が 1 枚に同居していて、用語の切り分けで止まりやすい
- **対策**: 図を先に指して「上が API、下がジョブ、真ん中で合流」を最初に言ってから、各経路の説明に入る。用語の切り分けは図で見せ、口では「2 つの入口」とだけ言う
- **キラーフレーズ**: 「interface を切ったから、同じ service を別の入口から再利用できる」

### スライド 12 (task-dispatcher 4 段絵 ★)

- **詰まる理由**: 4 つの箱 (Cloud Scheduler / task-dispatcher / Cloud Tasks / unified-api) と、各箱の役割を間違えやすい
- **対策**: 「① タイマー、② 振り分け係、③ 待ち行列、④ 働く係」の番号 + 役割の組を先に丸暗記。GCP の名前は後から付ける
- **キラーフレーズ**: 「Polaris は同じ『時間で起動する』を 4 段に分解しただけ」
