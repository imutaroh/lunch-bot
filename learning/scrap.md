## 2026-05-07 Day 2 Loop 4 完走 (interface 攻略 + RunAnnounce)

### このループでの最大の発見

**interface の "因果関係" を取り違えてた**。「interface を定義したら必ず実装しなきゃダメ」と思ってたが、正しくは「**interface を使う場所がある → そこに渡す値の型が満たす必要がある**」。
→ interface 単体は何も強制しない。「使う場所 (フィールド/引数)」とセットで初めて意味が出る。

### Loop 4 で押さえた4学び目標

1. **冪等性チェック** — `msgs[0]` の prefix を見て「もう発表済み?」を判定 → スキップで何もしない
2. **prefix判定** — 投稿の決まり文句を **定数化** して、投稿側と判定側で共有 (1箇所で書けば食い違いゼロ)
3. **bot除外** — recruit で bot 自身が🍱を押すから、参加者集計から外す必要 (`excludeUser`)
4. **早期return** — 異常系 (msgs 0件 / recruit nil) と "正常な特殊ケース" (3人未満 → お休み) を冒頭で処理 → メインロジック平坦

### 引っかかった所 (notes/05_service.md (e)〜(g) に追記)

- (e) interface の因果: "使う場所" → "実装が要る" の順
- (f) `make([]string, 0, len(users))` の第3引数 = 容量 = 最大値の事前予測 (= 拡張コピー回避)
- (g) `msgs` の正体 = Slack の HTTP スナップショット (= ETL の Extract フェーズ)

### 比較読みの効果

「自分なら announce はどう書くか」を3ステップで書いた → 実物は12ステップ。**差分は "取得フェーズ" の中に6ステップ詰まってる** (WhoAmI / 最近投稿 / 0件チェック / 冪等性 / 募集探し / 参加者取得)。  
**「人数取得」と一括りにしてた中に、業務固有の判定が複数潜んでる** ことを発見。これがこのループの最大の学び。

## 2026-05-07 Day 2 Loop 5 完走 (shuffler.go + テスト)

### 比較読みでの発見

- 自分の仮説: **テーブル方式** (人数ごとに分け方を事前定義、16人まで)
- 実物: **公式方式** `(n+2)/4` で算出 (n に上限なし)
- 学び: 「公式が動くなら 8行のコードでテーブル16ケースを置き換えられる」 = 抽象化の力

### 押さえた3学び目標

1. **math/rand** — `rand.Shuffle(n, swap)` の "関数を渡す" パターン (= 高階関数)。型に依らず汎用的に動く理由
2. **スライス分割** — `s[start:end]` は end を含まない、メモリは共有 (= 軽い)
3. **テーブル駆動テスト** — `for n := 3; n <= 20; n++` で全範囲を1つのテストで網羅確認

### 実験ログ (.claude/tmp/shuffle_count.go)

`rand.Shuffle` がスワップを何回呼ぶかを数えるスクリプトを作成して実行。結果: **n人なら swap が n-1 回呼ばれる** (Fisher-Yates の特性、線形時間 O(n))。「シャッフルが遅い」が起きにくいことを体感。

### 引っかかった所 → 06_shuffler.md (a)〜(d) に記録

- (a) `rand.Shuffle` がなぜ関数を引数で取るのか (= 型に依らず動かす仕組み)
- (b) Fisher-Yates の swap 回数 (実験で確認)
- (c) スライス分割記法 `s[start:end]` の意味と慣習
- (d) 公式 `(n+2)/4` がなぜ "四捨五入" として機能するか

### 設計判断の発見

- **テーブル方式 vs 公式方式** のトレードオフ → 公式方式は保守ゼロで n に上限なし、ただし「式の妥当性」は人間の直感では判断しづらい → **テストで担保** することで「式の正しさ」を機械的に証明している
- "計算" (`Shuffle`) と "表現" (`buildAnnouncement`) の **分離設計** → 単独テストが可能になる典型例

## 2026-05-07 Day 2 破壊テスト: bot 除外を消したらどうなる?

### 実験

`lunch_service.go:110` の `users = excludeUser(users, botID)` を1行コメントアウトして `cmd/simulate -n 4` を実行。

### 観察結果

```
正常時 (excludeUser あり):
グループA: <@U003> <@U001> <@U002> <@U004>
参加者 4 人 (bot自身を除外後)

破壊時 (excludeUser なし):
グループA: <@U004> <@U003> <@Ufakebot> <@U001> <@U002>
                            ^^^^^^^^^
                            ★ bot 自身が混ざってる
参加者 5 人 (bot自身を除外後)  ← ログは嘘
```

### 教訓

1. **bot 除外がないと bot 自身が参加者として混入** → 本物の Slack なら bot に @ メンションが飛ぶ (= 不毛な通知)
2. **グループサイズが狂う** → 想定4人グループが5人に
3. **ログ文言が嘘になる** → 「(bot自身を除外後)」と書いてあるのに実際には除外されてない。コードを変えるとログ文言が実態を反映しなくなる典型 = **「ログを信じる前にコードを確認する」習慣の重要性**
4. **interface の威力体感**: 本物Slackを叩かずに `cmd/simulate` で安全に検証できた。実Slackを汚さず、テストできる仕組み = Day 2 復習で学んだ interface の真の御利益

### 復元確認

`excludeUser` の行を戻して `simulate -n 4` を再実行 → bot が除外され、参加者4人で正常動作することを確認 → コードの状態は元通り。
