両方あると、**結論だけじゃなく "なぜそう結論したか" を未来の自分に渡せる**。

## 2026-04-30 デイリーレビュー (Day 1)

### 今日学んだこと Best 3
1. **流れを理解する** — コードを点ではなく「どう繋がって動くか」で読めるようになった (Slack → service → repository → Slack の往復)。これがめちゃくちゃ楽しかった
2. **依存性注入 (DI)** — `cmd/bot/main.go` の "プラモデル組み立て"。repo を svc に挿し、svc を handler に挿す順序の意味
3. **handler / service / repository の役割分担** — 3層それぞれが「何をして、何をしないか」が言語化できるようになった

### 次に潰したい課題 Top 3
1. **より詳しいコードリーディング** — 全体構造は掴めたが、各行のミクロをまだ精密に追えてない
2. **メソッド** — レシーバ `(s *LunchService)` の意味、ポインタ vs 値、メソッドと関数の境界
3. **インターフェイス** — 抽象化の核。「契約」と読み替える理屈はわかったが、まだ完全には腑に落ちてない

### まだ "なんで？" な疑問 3つ
→ 課題 Top 3 とほぼ同じ。「なぜそう書くか」が言葉で説明できないところが、そのまま潰したい課題と一致している。Day 2-3 でこの3点を中心に深掘る。
