## 1. 「言語化」3レベル定義

例: `users = excludeUser(users, botID)` の1行に対して。

| レベル | 発話例 | 何ができてる? |
|---|---|---|
| L1: 音読 | 「users から botID を除外して再代入」 | コードを日本語に変換 |
| L2: 翻訳 | 「参加者リストから bot 自身を取り除く」 | **何をしているか** |
| **L3: 問い直し** | 「bot は recruit で🍱を1個押す仕様 (ファーストペンギン回避) だから announce で除外が必要。消すと bot が参加者に混入してグループサイズ狂い + ログが嘘になる (Day 2 破壊テストで観察済み)」 | **なぜ書いたか / 消すと何が壊れるか** |

**社内勉強会レベル = L3**。Q&A の「即答」も L3 で出せること。

## 3. B. 実装可能性 (4項目) — B3 を最優先に格上げ

| # | 課題 | 制限時間 | 合格基準 | 重点 |
|---|---|---|---|---|
| B1 | `shuffler.go` + `shuffler_test.go` を白紙から再実装 | 45分 | `go test` 通過 | 通常 |
| B2 | `shuffler.go` まるごと削除 → 自力復元 | 60分 | 復元コードが `go test` 通過 + 壊れた連鎖を L3 で説明 | **任意** (Polaris優先で時間切れなら捨てる) |
| **B3** | **「水曜リマインド」サブコマンド追加** (3層に1メソッドずつ) | 60分 | `cmd/simulate` で動作確認 | **★最優先** (デモ動画を撮ってスライドに埋める) |
| B4 | Claude が故意に1行壊したコードのバグ特定 + 修正 | 30分 | 原因を L3 で説明できる | 通常 |

B1 = 記憶+理解 / B2 = 連鎖崩壊体感 (任意) / **B3 = 業務で書けるアピール** / B4 = デバッグ力。

**B3 がなぜ最優先か**: 笹本さんへの「業務に入れる状態」アピールに直結。「3層 + DI で組まれたアプリに、自分で 1 機能追加できる」を**デモ動画**で見せれば、説明 100 秒分の価値がある。

### 測定手順

#### B1: shuffler 白紙再実装 (45分)
1. 元コード (`internal/service/shuffler.go` + `shuffler_test.go`) を **10分だけ眺める** (おさらい)
2. `/tmp/shuffler_test/` に空ファイルを作る
3. 45分タイマー開始
4. 書き終わったら `go test` 実行
5. 合格: テスト通過 / 不合格: 落ちたら原因を1行 notes に追記

#### B2: shuffler.go 削除 → 自力復元 (60分)
1. 出題: `internal/service/shuffler.go` をまるごと削除
2. 60分タイマー開始
3. ステップ:
   - (a) 5分: 削除して `go build` 実行 → エラーメッセージを全部読む
   - (b) 5分: どこから連鎖して壊れたかを紙に図示
   - (c) 45分: 自分の頭で復元コードを書く
   - (d) 5分: `go test` + `cmd/simulate` で動作確認
4. 終わったら `git diff` で本物と自分の復元を比較 → 差分を notes に追記
5. 合格: 復元コードが動く + 「壊れた連鎖」を L3 で説明できる

#### B3: 水曜リマインド拡張 (60分)
1. 課題定義 (Claude が出題): 「水曜09:00 に発表されたグループに `🔔 今日のランチ忘れずに!` と投げる `remind` サブコマンドを追加せよ」
2. 60分タイマー開始
3. 仕上がったら `cmd/simulate` で動作確認
4. 合格: 3層 (handler / service / repository) すべてに1メソッドずつ正しく差し込めて動く

#### B4: バグ特定 (30分)
1. Claude が事前にコードを1行壊して別ブランチに置く (どこを壊したかは言わない)
2. 30分タイマー開始
3. 合格: 原因特定 + 修正 + L3 で説明できる (「なぜそれが原因か / 直すと何が直るか」)
