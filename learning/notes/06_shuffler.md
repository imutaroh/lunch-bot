### Q1. ご主人様なら **N人のリストを「3〜5人組のグループ」に分ける処理** をどう書く?

それだったら、**人数ごとにどんなグループに分けるかを考えます**。

具体例としては、以下の通りです：
1. 6人の場合：3人と3人
2. 8人の場合：4人と4人
3. 7人の場合：4人と3人

アプリのユーザー上の条件があるので、16人までのグループに分けて考えます。

→ **テーブル方式の発想**。各人数ごとに分け方を事前定義する。

### Q2. ランダム化はどうする?

ランダム化の部分については、パッケージをインポートすれば使えるツールがありそうなので、それを利用するのが良さそうです。実装としては、マップなどで正しくユーザーを取得して番号を割り振り、その上でランダムに並び替えるといった処理が必要になるのかもしれません。

→ Go 標準ライブラリ (`math/rand`) を使う、という方向性は正解。

### 3. 主な関数 / メソッド (引数・戻り値・1行説明)

| 名前 | 引数 | 戻り値 | 1行説明 |
|---|---|---|---|
| `Shuffle` | `userIDs []string` | `[][]string` | 入力をコピーしてシャッフル → サイズ通りに切り分ける |
| `decideGroupSizes` | `n int` | `[]int` | n人を3〜5人組にしたときの各グループ人数を返す |

#### `Shuffle` の中身 (16-35行目) のステップ

```
1. 早期return: len(userIDs) < 3 なら nil を返す
   ・呼び出し側 (RunAnnounce) でフォールバック (= "お休み" 投稿) する責任

2. コピーを作る: shuffled := make([]string, len(userIDs)); copy(shuffled, userIDs)
   ・元の userIDs を破壊しない (= 副作用ゼロ)
   ・excludeUser と同じ思想 ("入力を変更せず、新しい値を返す")

3. ランダム並び替え: rand.Shuffle(len(shuffled), func(i, j int) { ... })
   ・Fisher-Yates アルゴリズム (n人なら swap が n-1 回呼ばれる)
   ・第2引数の関数で「スライスの i, j を入れ替えるやり方」を教える

4. グループサイズ算出: sizes := decideGroupSizes(len(shuffled))
   ・例: 11人なら sizes = [4, 4, 3]

5. 切り分け: for _, size := range sizes { groups = append(groups, shuffled[start:start+size]); start += size }
   ・shuffled[start:start+size] = "start から size個ぶん" を切り出す
   ・先頭から順番に消費するので start を進めていく

6. return groups
```

#### `decideGroupSizes` の中身 (41-60行目)

```go
func decideGroupSizes(n int) []int {
    if n < 3 { return nil }

    numGroups := (n + 2) / 4    // グループ数 (4人組基準で四捨五入)
    base := n / numGroups        // 全グループの最低人数
    extra := n % numGroups       // この数だけ +1人のグループを作る

    sizes := make([]int, numGroups)
    for i := 0; i < numGroups; i++ {
        if i < extra {
            sizes[i] = base + 1
        } else {
            sizes[i] = base
        }
    }
    return sizes
}
```

→ 公式 `(n+2)/4` がキモ。これで **n人を「ほぼ4人組」** に分けるグループ数が決まる。

#### 結果一覧 (3〜20人)

| n | グループ数 | サイズ |
|---|---|---|
| 3 | 1 | [3] |
| 6 | 2 | [3, 3] |
| 7 | 2 | [4, 3] |
| 8 | 2 | [4, 4] |
| 9 | 2 | [5, 4] |
| 10 | 2 | [5, 5] |
| 11 | 3 | [4, 4, 3] |
| 14 | 3 | [5, 5, 4] |
| 15 | 4 | [4, 4, 4, 3] |
| 20 | 5 | [4, 4, 4, 4, 4] |

→ どのケースも **3〜5人の範囲に必ず収まる**。

### 4. 自分が引っかかった所

#### (a) `rand.Shuffle` がなぜ関数を引数で取ってるか

最初「なんで `func(i, j int)` を渡してるの?」が謎だった。

整理:
- `rand.Shuffle` のシグネチャ: `func Shuffle(n int, swap func(i, j int))`
- 第1引数: 要素数
- 第2引数: 「**i番目とj番目を入れ替えるやり方**」を教える関数

→ なぜこの形? **rand.Shuffle は "スライスを直接知らない"**。「ランダムなインデックスペアを生成する」担当だけ。「実際の入れ替え方」は呼び出し側が関数で渡す。

→ こうすれば `[]string` だろうが `[]int` だろうが `[]User` だろうが、**型に依らず動く** (= 高階関数の力)。

#### (b) Fisher-Yates アルゴリズムの swap 回数

実験して確認した: **n人なら swap が n-1 回呼ばれる**。

| n | swap 回数 |
|---|---|
| 3 | 2 |
| 5 | 4 |
| 8 | 7 |
| 10 | 9 |
| 20 | 19 |

→ 線形時間 O(n)。十分速い。

#### (c) スライス分割記法 `s[start:end]` の意味

```
s[start:end] = "start から end の手前まで" を切り出す
              (end は含まない)
```

例: `s = [A, B, C, D, E]`
- `s[0:3]` → `[A, B, C]` (インデックス 0,1,2)
- `s[2:5]` → `[C, D, E]` (インデックス 2,3,4)

ポイント: **end は含まない** (Go の慣習)。切り出しても **元のスライスのメモリを共有**してる (新しい配列を作らない = 軽い)。

#### (d) 公式 `(n+2)/4` がなぜ "四捨五入" になるか

整数除算 `/` は切り捨て。`+2` を入れることで `+0.5` 相当のオフセットになり、結果として四捨五入になる。

```
n=10 → (10+2)/4 = 12/4 = 3   ← 10/4=2.5 を四捨五入で 3 になる
n=11 → (11+2)/4 = 13/4 = 3   ← 11/4=2.75 を四捨五入で 3 になる
n=15 → (15+2)/4 = 17/4 = 4   ← 15/4=3.75 を四捨五入で 4 になる
```

これは **「4で割って四捨五入」のイディオム** (Goに限らず一般的)。

### 5. 他にもありえた選択 (Counter-design)

#### 案A: ご主人様の仮説どおり、人数別テーブルで実装する

```go
var groupTable = map[int][]int{
    3:  {3},
    4:  {4},
    5:  {5},
    6:  {3, 3},
    7:  {4, 3},
    8:  {4, 4},
    // ... 16人まで
}
```

却下理由:
- 16人を超えたら破綻 (社員が増えたらいちいち追加が必要)
- 公式なら **n に上限がない**
- 公式8行 vs テーブル16行以上 → 公式の方がシンプル

採用余地: 「3〜5人の範囲を厳密に守る」だけなら、テーブル方式の方が **見ただけで仕様が分かる** メリットはある。トレードオフ。

#### 案B: `Shuffle` を `[]string` ではなくジェネリクスで書く

Go 1.18+ ならジェネリクスで `func Shuffle[T any](items []T) [][]T` と書ける。

却下理由:
- lunch-bot では `[]string` (ユーザーID) しか使わない → 過剰汎用化
- "今必要な範囲で書く" という Go の哲学に合う

#### 案C: `decideGroupSizes` を Shuffle に inline する

却下理由:
- 関数を分けることで **`decideGroupSizes` を単独でテスト可能** に (= テーブル駆動テスト)
- 「グループ数の計算」と「ランダム化」を分離 → 関心分離

#### 案D: `rand.Seed` で乱数の種を固定する

`rand.Shuffle` は Go 1.20 以降、自動でランダムに初期化される (それ以前は固定 seed だった)。
昔のコードによくある `rand.Seed(time.Now().UnixNano())` は今は不要。

→ ここではあえて何もしてない。Go の進化を信じてる。

### 6. 次に読む人へのアドバイス (= 未来の自分へ)

1. **`rand.Shuffle(n, swap)` の "関数を渡す" パターンは Go の柔軟性の核**: interface と並んで「振る舞いを引数で渡す」高階関数の例。標準ライブラリで頻出。

2. **`copy` を見たら "副作用ゼロの作法" と読む**: 元のスライスを変更せず、新しいスライスで返す。`excludeUser` と同じ思想。Go の慣習。

3. **公式 `(n+2)/4` は「4で割って四捨五入」のイディオム**: lunch-bot 固有じゃない、汎用テクニック。覚えておくと整数演算で重宝する。

4. **"計算" と "表現" を分ける構造を意識する**: Shuffle (計算) → buildAnnouncement (表現) の役割分担。これを見ると「テストしやすいコード = 関心が分かれてるコード」が体感できる。

5. **テーブル駆動テストの良さは "網羅"**: `for n := 3; n <= 20; n++` で18ケースを1つの関数でカバー。仕様変更時に「全範囲が満たされてるか」を即時確認できる安心感。

6. **`TestShuffle_PreservesUsers` が示す Phase 2 の教訓**: 「ユーザーが消えない / 重複しない」という **不変条件** を明示的にテスト。シャッフル系で起きがちなバグの再発防止網。

7. **swap 回数 = n-1 を覚えておく**: Fisher-Yates の特性。n=100 でも swap 99回で済む線形時間。「シャッフルが遅い」は基本起きないと信じていい。

8. **早期return `if n < 3 { return nil }` が示す API の作法**: Shuffle は「3人未満を扱わない」と明示。呼び出し側 (`RunAnnounce`) が責任を持って "お休み" にフォールバックする設計分担。

9. **`shuffled[start:start+size]` の切り出しはメモリ共有**: 新しいスライスを作ってない (= 高速)。逆に言うと、`groups[0]` を編集すると元の `shuffled` も変わる。今回は読み取り専用なので問題ないが、書き換える設計に変えるなら注意。

10. **公式の "なぜ動くか" の証明はテストで担保してる**: `decideGroupSizes` の式を読んでも直感で正しさが分からなくても、テスト (`TestDecideGroupSizes_3to20`) が「全 n で 3〜5 を満たす」を保証してくれる。**「式が正しい」を人間が証明する代わりに、テストが証明する**。これも一流エンジニアのスタンス。
